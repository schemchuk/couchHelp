## Суть продукту (коротко)

**couchHelp — AI-orchestration layer для коучів по AVGS** (державне фінансування бізнесу в Німеччині). Не CRM — прошарок поверх існуючих інструментів коуча (WhatsApp, Bitrix24, Obsidian, Calendar).

Головна цінність: AI бере рутину, коуч контролює через approval-чергу.

Цільова аудиторія: AVGS-коучі в DACH-регіоні, три мови: DE / RU / UA.

## Ключові архітектурні рішення (не змінювати)

**1. audit_log — append-only**
Таблиця `audit_log` має тільки `INSERT`. Ніяких `UPDATE` і `DELETE`.
Це вимога аудиту DEKRA/TÜV для AVGS-провайдерів. Закладено в TypeScript типах.

**2. AI не відправляє без approval**
Фаза 1: кожне повідомлення проходить через кнопку коуча. Ніякої автоматичної відправки.

**3. Мова < 50 знаків → питаємо**
`detectLanguage()` повертає `null` для коротких повідомлень. Не вгадуємо DE/RU/UA.

**4. Монетизація — не реалізується зараз**
Поле `tier` в таблиці `tenants` присутнє, логіка обмежень не активна. Реалізувати після Фази 1.

**5. Не робимо алгоритмічну оцінку лідів**
Тільки ручні позначки коуча. Ніякого скорингу.

**6. Інтеграційний шар через адаптери**
Не "Bitrix-інтеграція" а "CRM-адаптер з реалізацією для Bitrix". Це дозволить масштабуватись на інші CRM.

## Схема БД (6 таблиць)

```
tenants         — коучі (tenant per coach)
clients         — ліди/клієнти коуча
messages        — повідомлення (inbound/outbound, всі канали)
audit_log       — append-only журнал всіх дій (ДЕКРА вимога)
promises        — обіцянки коуча клієнту з дедлайнами
templates       — шаблони повідомлень (DE/RU/UA, по сценаріях)
knowledge_items — база знань (stable/volatile)
```

Повна схема з полями — в `PROJECT_CONTEXT.md` розділ 6.
